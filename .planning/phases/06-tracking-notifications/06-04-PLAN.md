  <action>
Fix the RoleCardWithFilter component (lines 148-162) to correctly filter roles by status.

The current bug: `shouldShow()` checks `roleStatus?.status === statusFilter`. This fails in two cases:

1. When `roleStatus` data hasn't loaded yet (query in-flight), `roleStatus` is `undefined`, so `undefined?.status` is `undefined` and the card is hidden.
2. When role has no status set, API returns `{ status: null }`, so `null === 'save'` is false and the card is hidden -- this is CORRECT behavior (unstatused roles should NOT show under specific filters).

The real fix requires handling the loading state gracefully. When the useRoleStatus query is still loading, the card should NOT be hidden prematurely. Update the component to:

1. Destructure `isLoading` from `useRoleStatus(job.jobId)` in addition to `data`.
2. Update `shouldShow()` logic:
   - If `statusFilter === 'all'`, return `true` (unchanged).
   - If the status query is still loading, return `true` (show card while loading to avoid flash-of-empty, the card will hide/show once data arrives).
   - Otherwise, compare `roleStatus?.status === statusFilter` (unchanged).

This ensures:

- "All" shows everything (existing behavior, works).
- Specific filters show roles with that status once data loads.
- Cards are not prematurely hidden during the initial status fetch.
- Roles with null status (untracked) are correctly hidden from specific filters once data loads.

Do NOT change the useRoleStatus hook, the filters component, or the RoleCard component. The fix is isolated to the RoleCardWithFilter component's shouldShow logic and destructuring.
</action>
