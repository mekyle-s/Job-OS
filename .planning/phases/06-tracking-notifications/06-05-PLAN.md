Root cause: TanStack Query v5 default gcTime is 5 minutes. When user navigates away from queue page, the useMatchQueue component unmounts and gcTime countdown begins. After 5 minutes, cache is garbage collected. On return, no cached data exists and queue appears empty.

Fix three files:

1. **src/lib/hooks/use-match-queue.ts** - In the useQuery options object:
   - Add `gcTime: 1000 * 60 * 30` (30 minutes) to prevent garbage collection during normal navigation. This is the critical fix -- keeps queue data cached even when user browses other pages.
   - Add `refetchInterval: 1000 * 60 * 10` (10 minutes) to enable automatic background polling. This ensures queue data stays fresh without requiring manual poll retrigger. The interval is generous (10 min) to avoid unnecessary API load while still keeping data reasonably current.
   - Keep existing `staleTime: 1000 * 60 * 5` (5 minutes) unchanged.

2. **src/lib/hooks/use-role-status.ts** - In the useRoleStatus useQuery options object:
   - Add `gcTime: 1000 * 60 * 30` (30 minutes) to match the queue hook's caching behavior. Role status data should persist as long as queue data does, since they are displayed together. Without this, individual role status caches would still expire at 5 minutes, causing status to appear as null on return.
   - Keep existing `staleTime: 1000 * 60 * 5` (5 minutes) unchanged.
   - Do NOT modify useUpdateRoleStatus -- it handles mutations and does not need gcTime changes.

3. **src/app/providers.tsx** - In the QueryClient defaultOptions.queries object:
   - Add `gcTime: 1000 * 60 * 30` (30 minutes) as a global default. This ensures any other queries in the app also benefit from extended cache persistence. This is a safety net -- the hook-level gcTime values take precedence for queue and status queries.
   - Keep existing `staleTime` and `refetchOnWindowFocus` unchanged.

Do NOT change any other query options or hook behavior. This is a configuration-only change -- no logic changes needed.
</action>
