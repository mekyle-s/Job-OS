**Create `src/lib/matching/similarity.ts`:**

Implement `findSimilarEvidence(requirementEmbedding: number[], userId: string, topK?: number, minSimilarity?: number)`:
- Uses Drizzle ORM's `cosineDistance` helper (import from `drizzle-orm`) to compute similarity as `1 - cosineDistance(evidenceItem.embedding, requirementEmbedding)`
- Filters by userId AND non-null embeddings AND similarity > minSimilarity (default 0.3, kept low to avoid missing valid matches — LLM refinement handles precision)
- Orders by similarity DESC, limits to topK (default 10)
- Returns array of `{ id, title, company, content, metadata, similarity }`
- Handle edge case: if requirementEmbedding is empty or user has no evidence with embeddings, return empty array

Reference the research code example in 05-RESEARCH.md Pattern 1 and the Drizzle vector similarity guide. Use `sql` template literal for the similarity expression and `desc()` for ordering.

**Create `src/lib/matching/mapper.ts`:**

1. `mapEvidenceToRequirement(requirementId: string, userId: string)`:
   - Stage 1: Get the requirement from DB (with its embedding). If no embedding, generate one using `generateEmbedding` from embedder.ts and save it to DB.
   - Call `findSimilarEvidence` to get top-10 candidates.
   - Stage 2: For each candidate, call OpenAI LLM to validate the match. Use `gpt-4o-2024-08-06` with Structured Outputs (zodResponseFormat). The validation schema should output: `isMatch` (boolean), `supportingExcerpt` (string|null), `confidence` (number 0-1), `reasoning` (string).
   - System prompt: Conservative evidence validator. Only mark isMatch=true for clear, specific proof. Be conservative with confidence (0.8+ only for explicit matches). See 05-RESEARCH.md Pattern 1 for exact prompt.
   - Return only validated matches (isMatch=true), sorted by confidence DESC.
   - Use lazy OpenAI client init (per DEV-014).

2. `runMatchingForJob(jobId: string, userId: string)`:
   - Get all requirements for the job from DB.
   - Delete existing automated mappings for this job (call `deleteAllMappingsForJob` to clear stale mappings before re-matching — but preserve manual mappings where `isManual=true`).
   - For each requirement, call `mapEvidenceToRequirement`.
   - For each validated match, call `createEvidenceMapping` to persist.
   - Return summary: { totalRequirements, mappingsCreated, gaps }.
   - Process requirements sequentially (not parallel) to avoid OpenAI rate limits.

IMPORTANT: Use `zodResponseFormat` from `openai/helpers/zod` for Structured Outputs. Use .nullable() not .optional() for optional fields in the LLM response schema (per DEV-018).

**Create `src/lib/db/queries/match-queue.ts`:**

Implement `getRankedJobsQuery(userId: string, limit?: number)`:
- Uses raw SQL via `db.execute(sql`...`)` for complex aggregation (Drizzle doesn't support all window functions needed).
- CTE (WITH clause) `job_stats` joins job → requirement → evidence_mapping to compute:
  - `total_requirements` (COUNT DISTINCT requirement.id)
  - `evidence_coverage` (COUNT DISTINCT evidence_mapping.id WHERE userId matches)
  - `avg_confidence` (AVG of confidence scores)
  - `days_old` (EXTRACT(DAY FROM NOW() - source_updated_at))
- Main query computes:
  - `fit_score` = evidence_coverage / total_requirements (0-1)
  - `freshness_score` = EXP(-0.1 * days_old) (exponential decay, lambda=0.1)
  - `quality_score` = avg_confidence
  - `composite_score` = 0.6 * fit + 0.3 * freshness + 0.1 * quality
- Filters: is_active=true, parse_status='completed', total_requirements > 0
- Orders by composite_score DESC, limits to 50 (or param)
- See 05-RESEARCH.md Pattern 2 for exact SQL template

Important: The evidence_mapping join must filter by userId so rankings are user-specific. Jobs may have requirements but no mappings for this user — those should still appear with 0% coverage.

**Create `src/lib/matching/ranker.ts`:**

1. `calculateFreshnessScore(daysOld: number): number` — exponential decay with lambda=0.1. Returns score 0-1 (1.0 = today, ~0.5 at 7 days, ~0.1 at 30 days).

2. `getRankedJobs(userId: string, limit?: number)`:
   - Calls `getRankedJobsQuery` for the SQL aggregation.
   - Transforms raw results into typed objects.
   - Computes `nextAction` based on coveragePercent: 0% → "Review requirements", <50% → "Build more proof", <80% → "Fill gaps", >=80% → "Apply now".
   - Returns typed array per RankedJobSchema from matching.ts.

**Create `src/lib/matching/gap-analyzer.ts`:**

Implement `generateRoleBrief(jobId: string, userId: string)`:
- Get job details from DB.
- Get all requirements for the job.
- Get all evidence mappings for this job+user (via `getEvidenceMappingsForJob`).
- Categorize requirements into `covered` (has at least one mapping) and `gaps` (no mappings).
- For covered requirements, include evidence details: evidenceId, confidence, supportingExcerpt, evidenceTitle.
- Calculate fitSummary: coveragePercent, totalRequirements, covered count, gaps count.
- Identify `recommendedEmphasis`: top 3 evidence items used most frequently across covered requirements.
- Sort gaps by priority: 'required' first, then 'preferred', then 'nice_to_have', then 'unknown'.
- Return RoleBrief object matching the RoleBriefSchema.

See 05-RESEARCH.md Pattern 3 for the gap analysis approach and data structure.
