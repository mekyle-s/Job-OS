<objective>
Add vector embedding columns to evidence and requirement tables, create evidence mapping schema with audit trail, and build the embedding generation utility.

Purpose: This is the data foundation for the matching engine. Without embeddings on evidence and requirements, semantic similarity search cannot work. Without the mapping table, there is nowhere to store requirement-to-evidence connections. This plan adds all schema + migration + embedding infrastructure needed before the matching pipeline can run.

Output: Database migration with vector columns + HNSW indexes, evidence_mapping and evidence_mapping_audit tables, Zod validation schemas for mappings, CRUD queries for mappings, and embedding generator utility.
</objective>

Modify `src/lib/db/schema.ts` to add:

1. **Import `vector` from `drizzle-orm/pg-core`** — Add to the existing import at the top. Note: Drizzle ORM supports the `vector` column type natively for pgvector. Check Context7 for exact import syntax if needed.

2. **Add `embedding` column to `evidenceItem` table:**
   ```
   embedding: vector('embedding', { dimensions: 1536 }),
   ```
   Add AFTER the `isManual` field. This column is nullable — embeddings are generated asynchronously. No default value.

3. **Add `embedding` column to `requirement` table:**
   ```
   embedding: vector('embedding', { dimensions: 1536 }),
   ```
   Add AFTER the `isManuallyEdited` field. Also nullable.

4. **Create `evidenceMapping` table** with these columns:
   - `id` text PK
   - `userId` text NOT NULL, references user.id (cascade delete)
   - `requirementId` text NOT NULL, references requirement.id (cascade delete)
   - `evidenceItemId` text NOT NULL, references evidenceItem.id (cascade delete)
   - `confidence` real NOT NULL (0.0-1.0)
   - `supportingExcerpt` text NOT NULL
   - `reasoning` text (nullable)
   - `isManual` boolean default false NOT NULL
   - `wasEdited` boolean default false NOT NULL
   - `originalConfidence` real (nullable, preserves system's original confidence if user edits)
   - `createdAt` timestamp with timezone, defaultNow, NOT NULL
   - `updatedAt` timestamp with timezone, defaultNow, NOT NULL

   Indexes: userId, requirementId, evidenceItemId, confidence. Use the same table definition pattern as existing tables (see evidenceItem, requirementAudit for reference).

5. **Create `evidenceMappingAudit` table** with:
   - `id` text PK
   - `mappingId` text NOT NULL, references evidenceMapping.id (cascade delete)
   - `userId` text NOT NULL, references user.id (cascade delete)
   - `action` text NOT NULL ('create' | 'update' | 'delete' | 'accept' | 'reject')
   - `beforeValue` jsonb (nullable)
   - `afterValue` jsonb (nullable)
   - `createdAt` timestamp with timezone, defaultNow, NOT NULL

   Indexes: mappingId, userId.

6. **Generate and apply migration:**
   - Run `npx drizzle-kit generate` to create migration SQL
   - The migration MUST also create HNSW indexes for the embedding columns. If Drizzle doesn't auto-generate HNSW indexes, manually append to the generated migration file:
     ```sql
     CREATE INDEX IF NOT EXISTS evidence_item_embedding_idx ON evidence_item USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);
     CREATE INDEX IF NOT EXISTS requirement_embedding_idx ON requirement USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);
     ```
   - Run `npx drizzle-kit migrate` to apply

IMPORTANT: The vector column type requires the pgvector extension. The Docker Compose already uses `pgvector/pgvector:pg16` image and the health check verifies the extension. But the migration may need `CREATE EXTENSION IF NOT EXISTS vector;` at the top if not already present — check existing migrations.

Use DEV-021 pattern: text columns with Zod validation for action field (not pgEnum).

**Create `src/lib/schemas/matching.ts`** with Zod validation schemas:

1. `EvidenceMappingSchema` — validates a complete evidence mapping record (all fields from table)
2. `CreateEvidenceMappingSchema` — for creating new mappings: requirementId, evidenceItemId, confidence, supportingExcerpt, reasoning (optional), isManual
3. `UpdateEvidenceMappingSchema` — for editing: confidence (optional), supportingExcerpt (optional), reasoning (optional)
4. `RankedJobSchema` — for ranked queue items: jobId, title, company, location, url, fitScore, freshnessScore, qualityScore, compositeScore, coveragePercent, evidenceCoverage, totalRequirements, avgConfidence, daysOld, nextAction
5. `RoleBriefSchema` — for role brief: jobId, title, company, fitSummary (coveragePercent, totalRequirements, covered, gaps), requirementMap (covered array, gaps array), recommendedEmphasis

Use Zod v4 syntax (already in project at ^4.3.6). Follow patterns from `src/lib/schemas/evidence.ts` and `src/lib/schemas/requirements.ts`.

**Create `src/lib/db/queries/evidence-mapping.ts`** with CRUD functions:

1. `createEvidenceMapping(data)` — insert mapping + create audit record in transaction. Use crypto.randomUUID() for IDs. Follow the atomic audit trail pattern from requirements.ts (transaction with mapping insert + audit insert).
2. `createManyEvidenceMappings(mappings)` — bulk insert for automated matching (array of mappings). Create single audit record per mapping.
3. `getEvidenceMappingsForRequirement(requirementId, userId)` — get all mappings for a requirement with evidence item details (title, company, content via join or subquery).
4. `getEvidenceMappingsForJob(jobId, userId)` — get all mappings for all requirements of a job. Returns grouped by requirement.
5. `updateEvidenceMapping(mappingId, userId, updates)` — update mapping fields (confidence, supportingExcerpt, reasoning). Set wasEdited=true, preserve originalConfidence if first edit. Create audit record.
6. `deleteEvidenceMapping(mappingId, userId)` — delete mapping + create audit record.
7. `getMappingAuditTrail(mappingId)` — get audit history for a mapping.
8. `deleteAllMappingsForJob(jobId, userId)` — remove all mappings for a job (used when re-running matching).

All queries must follow existing patterns: user-scoped, use `db` from `@/lib/db`, import schema tables.

**Create `src/lib/matching/embedder.ts`** with embedding generation:

1. `generateEmbedding(text: string): Promise<number[]>` — single text embedding using OpenAI text-embedding-3-large with 1536 dimensions. Use lazy initialization pattern for OpenAI client (per DEV-014: `getOpenAIClient()` function, not direct instantiation). Handle empty string edge case (return empty array or throw).
2. `generateBatchEmbeddings(texts: string[]): Promise<number[][]>` — batch embedding for multiple texts. Batch up to 2048 texts per API call per OpenAI limits.
3. Constants: `EMBEDDING_MODEL = 'text-embedding-3-large'`, `EMBEDDING_DIMENSIONS = 1536`

Follow DEV-014 pattern from Phase 3: lazy-initialize OpenAI client at runtime to avoid build failures when OPENAI_API_KEY is missing.
