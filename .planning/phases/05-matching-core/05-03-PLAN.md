  <action>
**First, install TanStack Query:**
```bash
npm install @tanstack/react-query
```

**Create `src/app/api/matching/run/route.ts`** — POST endpoint to trigger matching:
- Authenticate with `verifySession()` (per DEV-024)
- Parse request body: `{ jobId: string }` validated with Zod
- Call `runMatchingForJob(jobId, userId)` from mapper.ts
- Return JSON: `{ success: true, totalRequirements, mappingsCreated, gaps }`
- Handle errors: 401 (not authenticated), 400 (invalid body), 500 (matching failed)
- This is a potentially long-running operation (~10-30 seconds for 15 requirements). Return response when complete. Do NOT background this — user needs to see results.

**Create `src/app/api/queue/route.ts`** — GET endpoint for ranked match queue:
- Authenticate with `verifySession()`
- Call `getRankedJobs(userId)` from ranker.ts
- Return JSON: `{ queue: RankedJob[], count: number, generatedAt: ISO timestamp }`
- If no ranked jobs, return empty array (not error)

**Create `src/app/api/matching/[jobId]/brief/route.ts`** — GET endpoint for role brief:
- Authenticate with `verifySession()`
- Extract jobId from route params
- Call `generateRoleBrief(jobId, userId)` from gap-analyzer.ts
- Return JSON: the full role brief object
- Handle: 401, 404 (job not found), 500

**Create `src/app/api/matching/[jobId]/mappings/route.ts`** — GET mappings for a job:
- Authenticate with `verifySession()`
- Extract jobId from route params
- Call `getEvidenceMappingsForJob(jobId, userId)` from evidence-mapping.ts
- Return JSON: `{ mappings: [...], count: number }`

**Create `src/app/api/matching/[jobId]/mappings/[mappingId]/route.ts`** — PATCH/DELETE individual mapping:
- PATCH: Update mapping fields (confidence, supportingExcerpt, reasoning). Validate with UpdateEvidenceMappingSchema. Call `updateEvidenceMapping`. Return updated mapping.
- DELETE: Call `deleteEvidenceMapping`. Return `{ success: true }`.
- Both methods authenticate with `verifySession()`.
- Handle: 401, 400, 404, 500

Follow the exact pattern from existing API routes: `src/app/api/jobs/[id]/requirements/[requirementId]/route.ts` for param extraction, error handling, and response format. Use `verifySession()` (not requireUser) per DEV-024.

  <files>src/app/providers.tsx, src/app/layout.tsx</files>
  <action>
**Create `src/app/providers.tsx`** — Client-side providers wrapper:
- Mark as `'use client'`
- Create QueryClient with sensible defaults:
  ```typescript
  const queryClient = new QueryClient({
    defaultOptions: {
      queries: {
        staleTime: 1000 * 60 * 5, // 5 minutes
        refetchOnWindowFocus: true,
        retry: 1,
      },
    },
  });
  ```
- IMPORTANT: Create the QueryClient using useState to prevent re-creation on every render:
  ```typescript
  const [queryClient] = useState(() => new QueryClient({...}));
  ```
- Export `Providers` component that wraps children with `<QueryClientProvider client={queryClient}>`.
- This component will be used in layout.tsx to wrap the app.

**Modify `src/app/layout.tsx`:**
- Import `Providers` from `./providers`
- Wrap the existing `{children}` inside `<body>` with `<Providers>{children}</Providers>`
- This is the ONLY change to layout.tsx — do not modify any other existing content (fonts, metadata, tailwind classes, etc.)

IMPORTANT: The Providers component must be a client component (`'use client'`) because TanStack Query uses React context. But layout.tsx should remain a server component — only the Providers wrapper is client-side. This is the standard Next.js pattern for client providers in server layouts.
