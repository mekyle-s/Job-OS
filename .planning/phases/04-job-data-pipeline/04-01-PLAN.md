  <action>
Add five new tables to `src/lib/db/schema.ts` after the existing evidence tables:

1. **user_criteria** table:
   - `id: text('id').primaryKey()` (use crypto.randomUUID() pattern from DEV-008)
   - `userId: text('user_id').notNull().references(() => user.id, { onDelete: 'cascade' })`
   - `jobFunction: text('job_function')` — e.g., "Software Engineering"
   - `locations: jsonb('locations').$type<string[]>()` — e.g., ["San Francisco", "Remote"]
   - `visaRequired: boolean('visa_required')`
   - `targetCompanies: jsonb('target_companies').$type<string[]>().notNull()` — max 10-15 per user decision
   - `isActive: boolean('is_active').default(true).notNull()`
   - `lastPolledAt: timestamp('last_polled_at', { withTimezone: true })`
   - `createdAt`, `updatedAt` timestamps (same pattern as existing tables)
   - Indexes: `user_criteria_user_id_idx` on userId, `user_criteria_is_active_idx` on isActive

2. **raw_job_source** table (preserves original API responses):
   - `id: text('id').primaryKey()`
   - `source: text('source').notNull()` — 'greenhouse' | 'lever'
   - `sourceJobId: text('source_job_id').notNull()`
   - `rawData: jsonb('raw_data').notNull()` — complete source response
   - `fetchedAt: timestamp('fetched_at', { withTimezone: true }).notNull()`
   - Unique composite index: `raw_job_source_lookup_idx` on (source, sourceJobId)

3. **job** table (canonical normalized model):
   - `id: text('id').primaryKey()`
   - `source: text('source').notNull()` — 'greenhouse' | 'lever'
   - `sourceJobId: text('source_job_id').notNull()`
   - `title: text('title').notNull()`
   - `company: text('company').notNull()`
   - `location: text('location').notNull()`
   - `description: text('description').notNull()` — full HTML/markdown
   - `url: text('url').notNull()` — application URL
   - `postedAt: timestamp('posted_at', { withTimezone: true })`
   - `sourceUpdatedAt: timestamp('source_updated_at', { withTimezone: true }).notNull()` — from source API
   - `metadata: jsonb('metadata').$type<{ departmentName?: string; officeLocation?: string; employmentType?: string; remote?: boolean; visaSponsorship?: boolean; [key: string]: unknown; }>()`
   - `parseStatus: text('parse_status').notNull().default('pending')` — 'pending' | 'processing' | 'completed' | 'failed'
   - `parseCompletedAt: timestamp('parse_completed_at', { withTimezone: true })`
   - `parseError: text('parse_error')`
   - `isActive: boolean('is_active').default(true).notNull()` — for tracking job deletions from source
   - `firstSeenAt: timestamp('first_seen_at', { withTimezone: true }).defaultNow().notNull()`
   - `createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull()`
   - Indexes: unique composite `job_source_lookup_idx` on (source, sourceJobId), `job_company_idx` on company, `job_source_updated_at_idx` on sourceUpdatedAt, `job_parse_status_idx` on parseStatus, `job_is_active_idx` on isActive

4. **requirement** table:
   - `id: text('id').primaryKey()`
   - `jobId: text('job_id').notNull().references(() => job.id, { onDelete: 'cascade' })`
   - `category: text('category').notNull()` — 'technical_skill' | 'experience' | 'education' | 'soft_skill' | 'other'
   - `priority: text('priority').notNull()` — 'required' | 'preferred' | 'unknown'
   - `normalizedText: text('normalized_text').notNull()` — clean, standardized phrasing
   - `sourceText: text('source_text').notNull()` — verbatim from posting
   - `sourceSpan: text('source_span')` — context/section header
   - `reviewStatus: text('review_status').notNull().default('parsed')` — 'parsed' | 'needs_review' | 'unparsed'
   - `isManuallyEdited: boolean('is_manually_edited').default(false).notNull()`
   - `createdAt`, `updatedAt` timestamps
   - Indexes: `requirement_job_id_idx` on jobId, `requirement_category_idx` on category, `requirement_review_status_idx` on reviewStatus

   NOTE: Use plain text columns for category, priority, and reviewStatus instead of pgEnum. Per DEV-012 patterns and existing codebase convention, the project uses text columns with runtime Zod validation rather than database-level enums. This avoids migration complexity when adding new enum values.

5. **requirement_audit** table:
   - `id: text('id').primaryKey()`
   - `requirementId: text('requirement_id').notNull().references(() => requirement.id, { onDelete: 'cascade' })`
   - `userId: text('user_id').notNull().references(() => user.id, { onDelete: 'cascade' })`
   - `action: text('action').notNull()` — 'create' | 'update' | 'delete'
   - `beforeValue: jsonb('before_value')`
   - `afterValue: jsonb('after_value')`
   - `createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull()`
   - Indexes: `requirement_audit_requirement_id_idx` on requirementId, `requirement_audit_user_id_idx` on userId

**Create `src/lib/schemas/jobs.ts`:**

Zod schemas for the canonical job model:

- `CanonicalJobSchema` — validated canonical job shape with fields matching the job table
- `JobMetadataSchema` — schema for the metadata JSONB field
- `CreateJobSchema` — for inserting new jobs
- `JobFilterSchema` — for filtering jobs (by company, location, parseStatus, isActive)
- Export TypeScript types using `z.infer<>`

**Create `src/lib/schemas/requirements.ts`:**

Zod schemas for requirement extraction and editing:

- `RequirementCategorySchema` — z.enum for the 5 categories
- `RequirementPrioritySchema` — z.enum for the 3 priorities
- `RequirementReviewStatusSchema` — z.enum for the 3 review states
- `ExtractedRequirementSchema` — single requirement from LLM:
  ```typescript
  {
    category: RequirementCategorySchema,
    priority: RequirementPrioritySchema,
    normalized_text: z.string(),
    source_text: z.string(),
    source_span: z.string().nullable()  // Use .nullable() per DEV-018
  }
  ```
- `JobRequirementsSchema` — LLM extraction output:
  ```typescript
  {
    requirements: z.array(ExtractedRequirementSchema),
    extractionNotes: z.string().nullable()  // Use .nullable() per DEV-018
  }
  ```
- `UpdateRequirementSchema` — for user edits (partial: priority, normalizedText, category)
- `CreateRequirementSchema` — for manually adding requirements
- Export TypeScript types

**Create `src/lib/db/queries/user-criteria.ts`:**

User-scoped CRUD for criteria profiles:

- `getUserCriteria(userId: string)` — get active criteria for user (return first active, since V1 is one profile per user)
- `upsertUserCriteria(userId: string, data: {...})` — create or update criteria. If user already has criteria, update it. Use Drizzle's onConflict or manual check+insert/update.
- `getAllActiveCriteria()` — get all active criteria for cron polling (no user filter)
- `updateLastPolledAt(criteriaId: string)` — update lastPolledAt timestamp after polling

All queries use `eq(userCriteria.userId, userId)` for user-scoping pattern (consistent with evidence.ts).

**Create `src/lib/db/queries/jobs.ts`:**

Job CRUD operations:

- `upsertRawJobSource(data: {...})` — insert or update raw source record, use onConflict on (source, sourceJobId)
- `upsertJob(data: {...})` — insert or update canonical job, return { job, isNew, isUpdated } to determine if requirement extraction needed
- `getJobsForUser(companies: string[], options?: { limit, offset, isActive })` — get jobs for user's target companies, ordered by sourceUpdatedAt desc. Include a `_count` of requirements.
- `getJobWithRequirements(jobId: string)` — get single job with all its requirements, ordered by priority then category
- `updateJobParseStatus(jobId: string, status: string, error?: string)` — update parse status after extraction
- `markJobsInactive(source: string, activeSourceJobIds: string[])` — mark jobs NOT in the latest fetch as inactive (per Pitfall 4 from research)

**Create `src/lib/db/queries/requirements.ts`:**

Requirement CRUD with audit trail:

- `createRequirements(jobId: string, requirements: ExtractedRequirement[])` — bulk insert extracted requirements
- `getRequirementsForJob(jobId: string)` — get all requirements for a job
- `updateRequirement(requirementId: string, userId: string, updates: UpdateRequirementInput)` — update requirement fields, set isManuallyEdited=true, create audit trail entry (before/after values)
- `deleteRequirement(requirementId: string, userId: string)` — soft-delete or hard-delete requirement, create audit entry with action='delete'
- `createManualRequirement(jobId: string, userId: string, data: CreateRequirementInput)` — add user-created requirement with audit trail
- `getRequirementAuditTrail(requirementId: string)` — get audit history for a requirement

Use transactions where appropriate (update + audit insert should be atomic). Follow the pattern from `src/lib/db/queries/evidence.ts` for user-scoped queries.

IMPORTANT: Use `import { db } from '@/lib/db'` and schema imports from `@/lib/db/schema`. Use `crypto.randomUUID()` for all ID generation (consistent with DEV-008).
