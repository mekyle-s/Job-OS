**Create `src/app/dashboard/jobs/page.tsx`:**

Job listings page showing all jobs matching user's criteria.

Layout:

- Page title: "Jobs"
- If no criteria set: Show message "Set up your criteria to start monitoring jobs" with link to /dashboard/criteria
- If criteria set but no jobs: Show message "No jobs found yet. Jobs are polled hourly — check back soon!"
- If jobs exist: Show job cards in a list layout

Each job card shows:

- **Title** (clickable, links to `/dashboard/jobs/[id]`)
- **Company** name
- **Location**
- **Freshness indicator:**
  - If firstSeenAt within 24 hours: green badge "New"
  - If sourceUpdatedAt within 24 hours: blue badge "Updated"
  - Otherwise: gray text showing relative time "3 days ago"
- **Parse status indicator:**
  - `completed`: green dot or checkmark
  - `processing`: spinner/pulse animation
  - `pending`: gray dot
  - `failed`: red dot with "Parse failed" text
- **Requirement count** if parsed: "X requirements extracted"

Pagination: Load more button at bottom (offset-based via API query params).

Fetch jobs from GET `/api/jobs` with limit=20. Handle loading and error states.

**Create `src/app/dashboard/jobs/[id]/page.tsx`:**

Job detail page with requirements display and editing.

Layout — two sections:

**Section 1: Job Information**

- Title, company, location
- Posted date (formatted)
- Link to original posting (job.url) — opens in new tab
- Parse status badge
- Freshness indicator

**Section 2: Requirements**

- Requirements grouped by category (technical_skill, experience, education, soft_skill, other)
- Each category has a collapsible section header with count
- Each requirement shows:
  - **Normalized text** (editable — click to edit inline or via modal)
  - **Priority badge:** "Required" (red), "Preferred" (blue), "Unknown" (gray) — clickable to toggle
  - **Source text** in smaller/muted font below (always visible, never editable per user decision)
  - **Review status badge:** "Parsed" | "Needs Review" | "Unparsed"
  - **Edit button** — opens inline edit for normalized text and priority
  - **Delete button** — with confirmation dialog, removes requirement
  - If `isManuallyEdited`: show small "Edited" indicator

**Add Requirement button:**

- At bottom of requirements section
- Opens a form to add: category (dropdown), priority (dropdown), normalized text (input)
- Source text defaults to empty string for manual entries
- On submit, POST to `/api/jobs/[id]/requirements`

**Requirement editing flow:**

- Click edit on a requirement → show inline form with current values
- Change priority or text → PATCH to `/api/jobs/[id]/requirements/[requirementId]`
- Delete → confirmation dialog → DELETE to `/api/jobs/[id]/requirements/[requirementId]`
- All changes update UI optimistically, revert on error

Style with Tailwind following existing dashboard conventions. Use consistent card layouts, badges, and interactive elements.

IMPORTANT UI priorities from user decisions:

- Focus on clear review states and edit controls
- Show system status (parse status, last polled)
- Keep original source text visible for reference
- Simple states, not numeric confidence percentages
