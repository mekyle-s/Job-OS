  <action>
All API routes use the DAL pattern from DEV-011: `requireUser()` for authentication. Follow the existing pattern from `src/app/api/evidence/upload/route.ts` for request/response structure.

**Create `src/app/api/criteria/route.ts`:**

GET handler:

- Authenticate with `requireUser()`
- Call `getUserCriteria(user.id)`
- Return criteria as JSON (or null if none set)
- 200 with criteria data or `{ criteria: null }`

PUT handler:

- Authenticate with `requireUser()`
- Parse request body as JSON
- Validate with a Zod schema inline or imported:
  ```typescript
  const CriteriaInputSchema = z.object({
    jobFunction: z.string().nullable().optional(),
    locations: z.array(z.string()).nullable().optional(),
    visaRequired: z.boolean().nullable().optional(),
    targetCompanies: z.array(z.string()).min(1).max(15), // 1-15 companies per user decision
  });
  ```
- On validation failure, return 400 with error details
- Call `upsertUserCriteria(user.id, validated)`
- Return 200 with updated criteria

**Create `src/app/api/jobs/route.ts`:**

GET handler:

- Authenticate with `requireUser()`
- Get user's criteria via `getUserCriteria(user.id)`
- If no criteria, return 200 with `{ jobs: [], message: "Set up your criteria first" }`
- Parse query params: `limit` (default 20), `offset` (default 0), `active` (default true)
- Call `getJobsForUser(criteria.targetCompanies, { limit, offset, isActive: active })`
- Return 200 with `{ jobs, total, hasMore }`

**Create `src/app/api/jobs/[id]/route.ts`:**

GET handler:

- Authenticate with `requireUser()`
- Extract `id` from route params
- Call `getJobWithRequirements(id)`
- If job not found, return 404
- Return 200 with job detail including requirements array

All routes should return proper error responses:

- 401 for unauthenticated (handled by requireUser redirect, but API should return JSON)
- 400 for validation errors with Zod error messages
- 404 for not found
- 500 for unexpected errors (wrapped in try/catch)

NOTE: For API routes that return JSON errors (not page redirects), use `verifySession()` instead of `requireUser()` since requireUser redirects to /sign-in which doesn't make sense for API calls. Return `Response.json({ error: 'Unauthorized' }, { status: 401 })` if no session.

**Create `src/app/api/jobs/[id]/requirements/route.ts`:**

GET handler:

- Authenticate with `verifySession()` (return 401 JSON if no session)
- Get jobId from route params
- Call `getRequirementsForJob(jobId)`
- Return 200 with requirements array

POST handler (manually add a requirement):

- Authenticate with `verifySession()`
- Get jobId from route params
- Parse and validate body with `CreateRequirementSchema`:
  ```typescript
  {
    category: RequirementCategorySchema,
    priority: RequirementPrioritySchema,
    normalizedText: z.string().min(1).max(500),
    sourceText: z.string().default(''),  // Empty for manual entries
  }
  ```
- Call `createManualRequirement(jobId, user.id, validated)`
- Return 201 with created requirement

**Create `src/app/api/jobs/[id]/requirements/[requirementId]/route.ts`:**

PATCH handler (edit a requirement):

- Authenticate with `verifySession()`
- Get requirementId from route params
- Parse and validate body with `UpdateRequirementSchema`:
  ```typescript
  {
    category: RequirementCategorySchema.optional(),
    priority: RequirementPrioritySchema.optional(),
    normalizedText: z.string().min(1).max(500).optional(),
    reviewStatus: RequirementReviewStatusSchema.optional(),
  }
  ```
- On validation failure, return 400
- Call `updateRequirement(requirementId, user.id, validated)` — this creates audit trail
- Return 200 with updated requirement

DELETE handler:

- Authenticate with `verifySession()`
- Get requirementId from route params
- Call `deleteRequirement(requirementId, user.id)` — this creates audit trail
- Return 200 with `{ success: true }`

IMPORTANT per user decision:

- Users can remove wrong extractions (DELETE)
- Users can change priority required <-> preferred (PATCH)
- Users can add missing requirements (POST)
- Users can edit normalized phrasing (PATCH)
- Original source text must remain visible (sourceText is never modified, only normalizedText)
- All changes tracked in audit trail (handled by query layer)
